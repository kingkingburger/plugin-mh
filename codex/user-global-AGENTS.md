# ~/.codex/AGENTS.md — 개인 글로벌 가이드 (원민호)

> 이 파일은 `~/.codex/AGENTS.md` 로 복사하면 Codex가 모든 프로젝트에서 본 가이드를 자동 로드한다.
> 프로젝트별 `AGENTS.md` 가 있으면 프로젝트 규칙이 우선하고, 이 파일은 보조 규칙으로 작용한다.
>
> **설치**
>
> ```powershell
> Copy-Item codex/user-global-AGENTS.md $env:USERPROFILE\.codex\AGENTS.md
> ```
>
> ```bash
> cp codex/user-global-AGENTS.md ~/.codex/AGENTS.md
> ```

## 사용자 프로필

- **이름**: 원민호
- **역할**: 소프트웨어 엔지니어
- **환경**: Windows, PowerShell, Git Bash
- **선호 언어**: Python (with `uv`, `ty`), TypeScript (with `bun`)
- **응답 언어**: 한국어. 기술 용어, 코드 식별자, 명령어, 파일 경로는 원문 유지

## 소통 / 업무 스타일

- 먼저 결론이나 현재 상태를 짧은 문장으로 말하고, 필요한 배경과 이유를 이어서 설명한다.
- 방향이 명확한 작업은 자율적으로 끝까지 진행하고 결과와 검증만 보고한다.
- 큰 작업이거나 선택에 따라 결과가 달라지는 작업은 단계별로 확인받는다.
- 구현 결과가 달라지는 모호함은 작업 전에 질문한다. 사소한 세부사항은 합리적으로 가정하고 진행한 뒤 가정을 보고한다.
- 내부 용어가 과하게 섞이면 한국어 문장으로 풀어서 설명한다.

## 응답 규칙

- 설명은 한국어로, 코드 / 식별자 / 경로 / 명령어는 원문 그대로 쓴다.
- 한국어 정서법을 지킨다. 한국어와 특수 문자(예: ä, ö, ü, ç, ñ)를 ASCII로 치환하지 않는다.
- 작업 완료 후 생성하거나 변경한 파일은 전체 경로로 출력한다.
- 새 개인 정보, 선호도, 반복되는 프로젝트 패턴을 발견하면 메모리 업데이트를 제안한다. 실제 메모리 기록은 사용자가 명시적으로 요청했을 때만 한다.
- 사용자에게 보이지 않는 명령 출력은 최종 답변에서 핵심만 요약한다.

## 도구 사용 원칙

- 현재 세션에 실제로 제공된 도구만 사용한다. 문서에 적힌 도구 이름이 있어도 현재 도구 목록에 없으면 실행 가능한 대체 수단을 쓴다.
- Claude 전용 도구명은 참고용으로만 해석한다.
  - `AskUserQuestion`: Codex에서는 꼭 필요한 경우 자연어 질문으로 대체한다.
  - `Tasks` / `TaskCreate`: Codex에서는 `update_plan` 이 있으면 사용하고, 없으면 짧은 체크리스트로 대체한다.
  - `Explore agent`: 단순 검색은 직접 수행하고, 복잡한 독립 조사만 사용 가능한 서브 에이전트로 위임한다.
- named skill이나 plugin이 현재 세션에 노출되어 있으면 해당 지침을 우선 확인한다. 노출되어 있지 않으면 같은 의도를 직접 수행하고 한 줄로 한계를 알린다.

## 셸 / 터미널 선호

- Windows-native 명령은 PowerShell을 우선 사용한다.
- POSIX shell 동작이 필요할 때만 명시적 Git Bash 경로를 사용한다.
- 기본 `bash` 또는 `exec_command` 의 `shell: "bash"` 는 WSL 경로로 잡혀 실패할 수 있으므로 먼저 시도하지 않는다.
- WSL bash 경로 실패가 발생해도 반복 복구 문구를 출력하지 말고, 조용히 명시적 Git Bash 경로 또는 PowerShell로 전환한다.
- 검증 / 설치 스크립트는 가능하면 PowerShell용 `.ps1` 과 Git Bash에서 실행 가능한 `.sh` 경로를 함께 유지한다.
- 셸별 차이가 있는 명령은 실제 사용 셸 기준으로 다시 확인한다.

## plugin-mh 라우팅 힌트

`~/.codex/prompts/` 또는 현재 세션의 skills/tools에 해당 기능이 있을 때만 사용한다. 이 목록은 자동 실행 규칙이 아니라 사용자 의도를 해석하기 위한 라우팅 힌트다. 실행 가능 여부가 불확실하면 현재 제공된 도구를 확인하고, 없으면 직접 수행한다.

### 사고 정리

- 모호한 요구 → `/vague` 또는 `/clarify`
- 전략 사각지대, 가정 점검 → `/unknown`
- 관점 전환, 다른 방법, diminishing returns → `/metamedium`
- 목표 상향, 10x, stretch → `/moonshot`
- 경쟁형 장기 목표, Judge Packet → `/deep-goal-council`

### 의사결정 / 리뷰

- 기술 선택, A vs B, 뭐가 나을까 → `/tech-decision`
- 다관점 토론 → `/agent-arena`
- 문서 리뷰 → `/expert-review`
- 코드 리뷰 → `/code-review`
- 고치고 다시 보는 리뷰 루프 → `/review-loop`

### 개발 / 검증

- 테스트 먼저, TDD, RED GREEN → `/tdd`
- 프로젝트 문서 체계, 하네스 → `/harness`
- 하네스 / 스킬 꼬임, 규칙 충돌, 과도한 탐색 → `/find-pulp`
- 자동 커밋 → `/auto-commit`
- 실제 동작, E2E, 브라우저 테스트 → `/live-verify`
- AI 슬롭, deslop, 슬롭 정리 → `/ai-slop-cleaner`

### 스펙 / 실행 / 세션

- 깊이 있는 스펙 → `/ouroboros`
- 계획 실행 → `/ouroboros-run`
- 경량 마무리 → `/closing-lite`
- 풀 마무리 → `/session-closing`
- 작업 보고서 / 일일 보고서 → `/daily-report`
- 인생 계획 / 1년 방향 / 오늘 미션 → `/life-plan`
- YouTube 요약 → `/youtube-digest`
- YouTube 슬라이드 캡처 → `/youtube-slides`
- plugin-mh 저장소의 스킬 관리 → `/skill-manage`

## 일반 작업 원칙

### Kapathy 가드

코드 작성, 버그 수정, 리팩터링, 코드 리뷰, 구현 계획 실행, 저장소 변경처럼 비용이 생기는 작업 전에는 가볍게 다음을 확인한다.

1. 실제 목표가 무엇인지, 결과를 바꾸는 모호함이나 충돌이 있는지 확인한다.
2. 현재 문제를 푸는 가장 작은 해법을 고른다.
3. 요청받지 않은 기능, 추상화, drive-by refactor를 섞지 않는다.
4. 변경 줄이 사용자 요청이나 그 요청으로 생긴 정리에 직접 연결되는지 확인한다.
5. 완료 전에 관찰 가능한 success criteria와 검증 방법을 정한다.

이 가드는 실행을 늦추기 위한 의식이 아니다. 단순하고 명확한 작업은 짧게 적용하고 바로 처리한다. 구현 결과가 달라지는 모호함이나 위험한 범위 확장이 보이면 먼저 좁히고 필요한 경우 반박한다.

### 모호함 처리

- 선택에 따라 구현 결과가 달라지면 작업 전에 질문한다.
- 실행 결과가 크게 달라지지 않는 세부사항은 합리적으로 가정하고 진행한 뒤 가정을 보고한다.
- 질문할 때는 가능한 선택지를 구체적으로 제시한다.

### 검증

- 구현 후 변경 표면에 맞는 검증을 실행한다.
- 문서 변경: diff, 링크 / 경로, 인코딩, 설치본 동기화 여부를 확인한다.
- TypeScript 변경: `bun` 기반 테스트와 타입 체크를 우선한다.
- Python 변경: `uv`, `uv run ty check`, `pytest` 를 우선한다.
- Rust 변경: `cargo fmt`, `cargo clippy`, `cargo test` 를 우선한다.
- UI / E2E 변경: 가능한 경우 실제 화면이나 브라우저 표면에서 검증한다.
- 검증을 실행하지 못했으면 최종 답변에서 이유를 명확히 말한다.

### 문서 위치

- 반복 운영 규칙은 가장 좁고 권위 있는 문서에 둔다.
- 전역 `AGENTS.md` 는 사용자 선호와 공통 운영 원칙만 담는다.
- repo-specific 규칙은 해당 repo의 `AGENTS.md`, `CLAUDE.md`, `docs/` 또는 가까운 도메인 문서에 둔다.
- 전역 문서에 모든 세부 워크플로우를 누적하지 않는다.

### 메모리

- 사용자가 세션 마무리나 메모리 누적을 요청했고, 큰 변화 / 새 패턴 / 반복 선호가 있으면 메모리 업데이트를 제안한다.
- 실제 메모리 파일 갱신은 사용자가 명시적으로 요청했을 때만 한다.
- 하루 작업을 감사 가능한 문서로 남겨야 하면 `/daily-report` 의도에 맞춰 별도 Markdown 보고서를 만든다.

## 커밋 메시지 컨벤션

- 한국어로 작성한다.
- 접두사는 `추가:`, `수정:`, `삭제:`, `개선:` 중 하나를 사용한다.
- 형식은 `<접두사> <한 줄 요약>` + 빈 줄 + 상세 내용이다.
- 상세 내용에는 무엇을 왜 바꿨는지와 제외한 범위를 적는다.
- 검증 결과와 남은 위험은 커밋 메시지 본문에 넣지 않고 최종 답변에서 보고한다.
- AI 협업 커밋에는 `Co-Authored-By` 를 포함한다.
- `git add -A` 대신 변경 파일을 명시적으로 add한다.
- 명시 허락 없이 `--no-verify` 를 사용하지 않는다.

## 위임 / 에이전트 호출

- 단순 검색은 직접 도구 호출로 처리한다. 예: `rg`, 파일 읽기, git 조회.
- 복잡한 멀티스텝 조사나 독립 병렬 작업만 사용 가능한 서브 에이전트에 위임한다.
- 내가 작성한 코드는 먼저 직접 검증한다. 큰 변경이나 위험한 변경은 별도 리뷰 패스를 붙인다.
