# AI Slop Cleaner

AI가 생성한 "슬롭(slop)" 코드를 **스코프 드리프트 없이** 정리한다. 동작은 보존하면서 중복·죽은 코드·불필요 추상화·경계 위반·약한 회귀 커버리지를 제거하는 **경계가 명확한 정리 워크플로우**.

> 핵심 철학: **삭제 우선, 추가 신중**. 새 기능을 만드는 게 아니라 이미 있는 노이즈를 정리한다.

## When to Use

- 사용자가 `deslop`, `anti-slop`, `AI slop`, `슬롭 정리`라고 말할 때
- AI가 만든 코드가 "동작은 하지만 시끄럽고/반복적이고/과추상화됐다"고 느껴질 때
- 후속 구현이 중복 로직·죽은 코드·래퍼 레이어·경계 누수·약한 회귀 커버리지를 남겼을 때
- 사용자가 `--review` 옵션으로 리뷰어 전용 슬롭 점검을 요청할 때
- 목표가 신규 기능이 아닌 **단순화·정리**일 때

## When Not to Use

- 메인 작업이 신규 기능 빌드 또는 제품 변경일 때
- 점진적 정리가 아닌 **전체 재설계**를 원할 때
- 단순화 의도가 없는 일반 리팩토링 요청일 때
- 동작이 불명확해서 테스트나 명시적 검증 계획으로 보호할 수 없을 때

## Posture (작업 자세)

- 사용자가 명시적으로 동작 변경을 요청하지 않는 한 **동작 보존**
- 코드 편집 전에 **회귀 테스트로 동작을 락(lock)** 한다
- 코드 편집 전에 **정리 계획**을 먼저 쓴다
- **삭제 > 추가**. 코드 제거가 가능하면 제거부터 시도
- 새 유틸리티 도입 전 기존 유틸리티·패턴 재사용을 먼저 검토
- 사용자가 명시 요청하지 않는 한 **새 의존성 금지**
- diff는 작고 되돌릴 수 있게, 한 종류의 스멜에만 집중
- 간결하고 증거 밀도가 높은 보고
- 새 지시는 로컬 스코프 업데이트로만 적용 — 이전 충돌 없는 제약은 유지

## Scoped File-List Usage

명시적인 파일 리스트 또는 변경된 파일 범위로 정리 범위를 묶을 수 있다.

- 좋은 예: `/ai-slop-cleaner skills/ralph/SKILL.md skills/ai-slop-cleaner/SKILL.md`
- 좋은 예: 다른 워크플로우(예: review-loop, ouroboros-run)가 자신의 변경 파일만 넘기는 경우
- 짧은 파일 리스트라도 회귀-안전 워크플로우는 동일하게 적용
- 사용자가 명시 요청하지 않는 한 변경 파일 범위를 더 넓은 정리로 **묵시적 확장 금지**

## Review Mode (`--review`)

`--review`는 정리 작업이 초안화된 뒤 실행되는 **리뷰어 전용 패스**다. 슬롭 제거 작업에서 작성자(Writer) ↔ 리뷰어(Reviewer) 분리를 강제하기 위해 존재한다.

- **Writer pass**: 테스트로 동작을 락한 상태에서 정리 변경을 수행
- **Reviewer pass**: 정리 계획, 변경 파일, 검증 증거를 검사

한 패스가 동시에 수정과 자체 승인을 해서는 안 된다.

리뷰 모드에서는:

1. **편집 금지** — 파일을 수정하지 않는다
2. 정리 계획·변경 파일·회귀 커버리지를 검토
3. 다음을 특히 점검:
   - 남아 있는 죽은 코드 / 사용되지 않는 export
   - 통합되지 않은 중복 로직
   - 경계를 여전히 흐리는 불필요한 래퍼·추상
   - 보존된 동작에 대한 테스트 누락 / 약한 검증
   - 의도 없이 동작이 바뀐 정리
4. 리뷰어 판정(verdict)과 후속 조치 항목(required follow-ups) 작성
5. 필요한 변경은 별도의 Writer 패스로 핸드오프 — 한 패스에서 수정+승인 금지

## Workflow

### Step 1: 현재 동작 보호 (테스트 우선)

- 무엇이 유지되어야 하는지 식별
- **편집 전에** 가장 좁은 회귀 테스트를 추가/실행
- 테스트가 먼저 못 들어가면 **검증 계획을 명시적으로 기록**한 뒤 코드 변경

### Step 2: 정리 계획 작성 (코드보다 먼저)

- 요청된 파일/기능 영역으로 범위를 묶는다
- 제거할 **구체적인 스멜**을 나열
- **안전한 삭제 → 위험한 통합** 순서로 정렬

### Step 3: 슬롭 분류 (편집 전)

| 종류 | 정의 |
|------|------|
| **Duplication** | 반복 로직, 복붙된 분기, 중복 헬퍼 |
| **Dead code** | 사용되지 않는 코드, 도달 불가 분기, 오래된 플래그, 디버그 잔재 |
| **Needless abstraction** | 패스스루 래퍼, 사변적 간접 호출, 1회용 헬퍼 레이어 |
| **Boundary violations** | 숨겨진 결합, 잘못 배치된 책임, 잘못된 레이어 import/사이드 이펙트 |
| **Missing tests** | 락되지 않은 동작, 약한 회귀 커버리지, 엣지 케이스 누락 |
| **UI/design defaults** | AI가 만든 UI를 "검토되지 않은 느낌"으로 만드는 제너릭 비주얼 패턴 |

### Step 3.5: UI/Design 리뷰 체크리스트 (해당 시)

리뷰 프롬프트로 사용. 절대 금지가 아니라 의도된 브랜드·접근성·정보 밀도·디자인 시스템 선택은 존중한다.

- **한국어 가독성**: 본문 텍스트 11~12px 의심 — 한국어 본문은 보통 14px 이상 필요(검증된 고밀도 데이터 예외 제외)
- **그림자 절제**: 모든 표면·로고·배경·카드·아이콘에 box-shadow 의심 — elevation/인터랙션을 명확히 할 때만 유지
- **콘텐츠 위계**: 타이틀이 이미 메시지를 전달하면 eyebrow/title/description/추가 `<p>` 반복 제거. 제품 보이스가 아니면 제너릭 이모지 배지 제거
- **팔레트 근거**: 브랜드·시스템 근거 없는 기본 AI 블루/퍼플(예: Tailwind `#3B82F6`) 의심
- **레이아웃 리듬**: 리듬·강조·비대칭·캐러셀·벤토·카드 무게 차별화가 어울리는 맥락에서 과한 균등 3·4컬럼 그리드 의심
- **그라데이션 절제**: 브랜드가 의도적으로 그라데이션을 소유하지 않는 한 과한 그라데이션 톤다운

### Step 4: 한 번에 한 종류의 스멜만 (단일 패스 편집)

| Pass | 대상 |
|------|------|
| Pass 1 | **Dead code 삭제** |
| Pass 2 | **중복 제거** |
| Pass 3 | **네이밍·에러 처리 정리** |
| Pass 4 | **테스트 보강** |

- 각 패스 후 타겟 검증 재실행
- 같은 편집 세트에 서로 무관한 리팩토링 **번들링 금지**

### Step 5: 품질 게이트 통과

- 회귀 테스트 그린 유지
- 변경 영역에 해당하는 lint, typecheck, unit/integration 테스트 실행
- 정적 분석·보안 점검이 있다면 실행
- 게이트 실패 시 **이슈를 고치거나 위험한 정리를 되돌린다** — 강행 금지

### Step 6: 증거 밀도 높은 보고로 마무리

항상 다음을 보고:

- **Changed files** — 변경 파일 목록
- **Simplifications** — 제거된 스멜 종류와 위치
- **Behavior lock / Verification run** — 락한 테스트와 게이트 결과
- **Remaining risks** — 남아 있는 리스크와 후속 조치

## Usage

- `/plugin-mh:ai-slop-cleaner <대상>`
- `/plugin-mh:ai-slop-cleaner <대상> --review`
- `/plugin-mh:ai-slop-cleaner <file-a> <file-b> <file-c>`
- 다른 워크플로우(`review-loop`, `ouroboros-run`)에서 후속 정리 패스로 호출 가능

## Good / Bad Fits

**Good**

- `deslop this module: 너무 많은 래퍼, 중복 헬퍼, 죽은 코드`
- `src/auth 의 AI 슬롭 정리해줘 — 동작은 그대로 두고 경계만 조이기`
- `/ai-slop-cleaner skills/foo/SKILL.md skills/bar/SKILL.md --review`

**Bad**

- `auth 모듈을 SSO 지원하도록 리팩토링`  ← 신규 기능
- `포매팅만 정리해줘`  ← 슬롭 정리 의도 부재
- 동작 명세가 불명확해서 회귀 락 자체가 불가능한 코드 영역

## 다른 스킬과의 관계

| 상황 | 권장 |
|------|------|
| 코드 작성 직후 품질 검증 | `review-loop` (먼저) → 필요 시 `ai-slop-cleaner` |
| `ouroboros-run` 직후 잔여 슬롭 정리 | `ai-slop-cleaner` (변경 파일 스코프) |
| 동작 자체를 바꿔야 함 | `tdd` 사이클로 진입 |
| 요구사항이 모호함 | `vague` 또는 `clarify` 먼저 |
