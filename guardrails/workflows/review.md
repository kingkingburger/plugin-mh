# Review Workflow Guardrails

## 리뷰 우선순위

1. 실제 버그와 회귀
2. 보안, 권한, 민감정보 위험
3. 데이터 손상, 마이그레이션, 호환성 위험
4. 누락된 테스트와 검증 공백
5. 유지보수성과 가독성

스타일 취향은 위 항목보다 뒤에 둔다.

## 리뷰 방식

- findings를 먼저 쓴다.
- 각 finding은 파일/라인, 영향, 재현 조건, 수정 방향을 포함한다.
- 추측은 추측이라고 표시한다.
- 직접 작성한 코드는 독립 리뷰처럼 포장하지 않는다. 필요하면 별도 cold-read 패스를 둔다.

## 법칙 기반 체크

- Hyrum: 관찰 가능한 계약이 바뀌었는가?
- Gall: 작은 작동 시스템 없이 큰 구조부터 만들었는가?
- Goodhart: 지표를 맞추려고 가치 없는 테스트나 문서를 추가했는가?
- Kernighan: 디버깅하기 어려운 영리한 코드인가?
- YAGNI: 지금 필요 없는 확장성을 만들었는가?
- Technical Debt: 의도적 부채에 상환 계획이 있는가?

## 완료 기준

- 심각도 높은 finding이 먼저 보고됐다.
- 테스트 공백과 남은 위험이 분리되어 있다.
- 변경 요약은 findings 뒤에 온다.
