# /ai-slop-cleaner — AI Slop Cleaner: 회귀 안전 + 삭제 우선 정리 워크플로우

AI가 생성한 슬롭 코드를 동작 보존 + 삭제 우선 + 한 번에 한 종류씩 정리한다.

> Codex CLI 슬래시 커맨드로 호출되거나 자연어로 트리거된다. `$ARGUMENTS`에 사용자 입력이 주입된다.

```
[1] 동작 보호 (테스트 우선)
       ↓
[2] 정리 계획 작성
       ↓
[3] 슬롭 분류 (Dead / Dup / Abstract / Boundary / Test / UI)
       ↓
[4] 단일 패스 편집 (한 종류씩)
       ↓
[5] 품질 게이트
       ↓
[6] 증거 밀도 보고
```

핵심 원칙:
- **삭제 > 추가**. 새 기능을 만드는 게 아니다.
- 동작은 보존, 사용자가 명시 요청해야만 변경.
- 테스트로 동작을 락(lock) 한 다음에 편집.
- 한 편집 세트에 무관한 리팩토링 번들링 금지.
- `--review` 모드는 편집 금지, 리뷰어 판정만.

## When to Use

- 사용자가 "deslop", "anti-slop", "AI slop", "슬롭 정리", "쓰레기 코드 청소"라고 말할 때
- 신규 기능이 아니라 단순화·정리가 목표일 때
- 다른 워크플로우(`review-loop`, `ouroboros-run`)가 변경 파일을 넘겨 후속 정리를 요구할 때
- `--review`로 리뷰어 전용 패스를 요청할 때

## When Not to Use

- 신규 기능 빌드 / 동작 변경 / 전체 재설계
- 단순화 의도 없는 일반 리팩토링
- 동작 명세가 불명확해서 회귀 락 불가능한 영역

## Workflow

### Step 1: 동작 보호

- 보존해야 할 동작 식별
- 가장 좁은 회귀 테스트를 **편집 전에** 추가/실행
- 테스트가 안 되면 **검증 계획을 명시적으로 기록**한 뒤 편집

### Step 2: 정리 계획

- 범위 = 요청된 파일/기능 영역
- 구체적 스멜 나열
- 순서: **안전한 삭제 → 위험한 통합**

### Step 3: 슬롭 분류

| 종류 | 정의 |
|------|------|
| Duplication | 반복 로직, 복붙 분기, 중복 헬퍼 |
| Dead code | 미사용 코드, 도달 불가 분기, 오래된 플래그, 디버그 잔재 |
| Needless abstraction | 패스스루 래퍼, 사변적 간접 호출, 1회용 헬퍼 레이어 |
| Boundary violations | 숨겨진 결합, 잘못 배치된 책임, 잘못된 레이어 import/사이드 이펙트 |
| Missing tests | 락되지 않은 동작, 약한 회귀, 엣지 케이스 누락 |
| UI/design defaults | 제너릭 비주얼 패턴 — Step 3.5 체크리스트 참고 |

### Step 3.5: UI/Design 체크리스트 (해당 시)

리뷰 프롬프트로만 사용. 의도된 브랜드·접근성·디자인 시스템 선택은 보존.

- 한국어 본문 11~12px 의심 (보통 14px 이상)
- 모든 표면에 box-shadow 의심
- eyebrow/title/description 중복 위계 제거
- 근거 없는 AI 블루/퍼플 팔레트 의심
- 균등 3·4컬럼 그리드의 리듬 부재 의심
- 브랜드 의도 없는 과한 그라데이션 톤다운

### Step 4: 한 번에 한 종류의 스멜

| Pass | 대상 |
|------|------|
| Pass 1 | Dead code 삭제 |
| Pass 2 | 중복 제거 |
| Pass 3 | 네이밍·에러 처리 정리 |
| Pass 4 | 테스트 보강 |

- 각 패스 후 타겟 검증 재실행
- 무관한 리팩토링 같은 편집 세트에 번들링 금지

### Step 5: 품질 게이트

- 회귀 테스트 그린 유지
- 변경 영역의 lint / typecheck / unit / integration 실행
- 정적 분석·보안 점검 가능하면 실행
- 게이트 실패 시 **수정 또는 정리 롤백** — 강행 금지

### Step 6: 보고

- Changed files
- Simplifications (스멜 종류 + 위치)
- Behavior lock / Verification run
- Remaining risks + 후속

## Review Mode (`--review`)

`--review`는 편집 금지 모드다. 정리 작업이 이미 초안화된 뒤 실행한다.

1. 파일 편집 금지
2. 정리 계획·변경 파일·회귀 커버리지 검토
3. 점검 항목:
   - 남은 dead code / 미사용 export
   - 통합 안 된 중복
   - 경계를 여전히 흐리는 래퍼
   - 보존 동작에 대한 테스트 누락
   - 의도 없이 바뀐 동작
4. 리뷰어 판정 + 후속 조치 항목
5. 수정은 별도 Writer 패스로 핸드오프

## Output Format

```
## AI Slop Cleaner 결과

### 정리 계획 (요약)
- 범위: {파일 / 기능 영역}
- 분류: Dead code N, Duplication N, Needless abstraction N
- 순서: Pass 1 → Pass 2 → Pass 3

### 변경 파일
- {경로} — {요약}

### 단순화 내역
- [{종류}] {경로}:{라인} — {설명}

### 동작 락 / 검증
- 회귀 테스트: N/N 통과
- lint/typecheck: PASS|FAIL

### 남아 있는 리스크
- {리스크} → {후속 액션}
```

## Constraints

- 동작 보존 (사용자 명시 요청 시에만 변경 허용)
- 편집 전 회귀 락 (테스트 또는 명시 검증 계획)
- 삭제 우선, 추가 신중 — 새 의존성 금지(명시 요청 시 제외)
- 한 편집 세트 = 한 종류의 스멜 (번들링 금지)
- 변경 파일 스코프를 묵시적으로 확장 금지
- `--review`는 편집 금지 — 리뷰어 판정만
- 게이트 실패 시 롤백 또는 수정 — 강행 금지
