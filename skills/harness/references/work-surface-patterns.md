# Work Surface Patterns

Harness는 코드 프로젝트만을 위한 문서 생성기가 아니다. 먼저 작업 표면을 판별하고, 그 표면에서 반복될 입력, 산출물, 역할, 검증 기준, 갱신 리듬을 고정한다.

## 표면별 기본 하네스

| 작업 표면 | 입력 | 산출물 | 역할 | 검증 |
|-----------|------|--------|------|------|
| 기획 | 아이디어, 시장/사용자 메모, 제약, 목표 | PRD, 로드맵, 우선순위, 실험 설계, 의사결정 로그 | PM, 사용자 대변인, 리스크 분석가, 실행 플래너 | 목표-범위-비목표 일관성, 성공 지표, 1차 실험 가능성 |
| 문서작업 | 초안, 참고 자료, 독자, 배포 채널 | 목차, 스타일 가이드, 초안, 리뷰 체크리스트, 배포/갱신 규칙 | 에디터, 도메인 전문가, 팩트체커, 독자 대변인 | 독자 적합성, 출처 추적, 용어 일관성, 액션 가능성 |
| 디자인 | 사용자 여정, 브랜드 원칙, 기능 요구, 제약 | 사용자 흐름, 와이어프레임, 디자인 원칙, 컴포넌트/토큰 명세 | UX 리서처, 인터랙션 디자이너, 비주얼 디자이너, 접근성 리뷰어 | 핵심 여정 완주, 시각 위계, 접근성, 상태/엣지 케이스 |
| 리서치 | 질문, 후보 소스, 비교 기준 | 리서치 브리프, evidence ledger, synthesis memo, 결정 옵션 | 리서처, 회의론자, 종합자, 의사결정자 | 소스 품질, 반례 포함, 인용 가능성, 결론-증거 연결 |
| 운영 | 반복 업무, 일정, 책임자, 장애/예외 사례 | 런북, 체크리스트, SLA/SLO, 핸드오프, 모니터링 기준 | 운영자, 품질 관리자, 에스컬레이션 담당, 회고 진행자 | 재현 가능성, 예외 처리, 책임 경계, 실패 후 복구 |
| 개발 | 코드, 이슈, 아키텍처, 테스트/배포 흐름 | AGENTS.md, ARCHITECTURE.md, guardrails/, 테스트/릴리즈 절차 | 아키텍트, 구현자, 리뷰어, QA | 빌드/테스트, 경계 규칙, 회귀 방지, 릴리즈 가능성 |

## 선택 규칙

1. 사용자가 "기획", "문서", "디자인", "리서치", "운영"을 말하면 코드 구조보다 해당 표면의 입력/산출물/검증부터 묻는다.
2. 여러 표면이 섞이면 가장 앞 단계의 불확실성을 기준으로 시작한다. 예: 디자인이 막혔지만 PRD가 없으면 기획 하네스를 먼저 만든다.
3. 산출물은 파일명보다 운영 규칙이 중요하다. 누가 언제 갱신하고, 무엇을 보고 완료로 판단하는지 반드시 포함한다.
4. 기존 자료가 있으면 삭제하거나 덮어쓰지 말고, 빈 부분과 drift만 보완한다.
5. 검증은 "파일 존재"가 아니라 실제 사용 시나리오 기준이어야 한다.

## 공통 산출물 골격

- `AGENTS.md` 또는 루트 가이드: 작업 목적, 금지 사항, 반복 절차, 검증 명령/체크리스트
- `docs/<surface>/index.md`: 표면별 문서 진입점
- `docs/<surface>/templates/`: 반복 산출물 템플릿
- `docs/<surface>/decisions.md`: 결정과 근거
- `docs/<surface>/review-checklist.md`: 승인 전 확인 기준
- `_workspace/`: 에이전트 중간 산출물과 evidence
