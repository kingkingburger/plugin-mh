# Team Blueprints

Deep Goal Council은 같은 입력을 여러 경쟁 팀이 서로 다른 철학으로 해석하게 한다. 각 팀은 계층 구조를 갖지만, 실제 런타임이 중첩 팀을 지원하지 않으면 역할 라운드로 평탄화한다.

## 역할 계층

| 계층 | 역할 | 책임 |
| --- | --- | --- |
| Council Chair | 전체 총괄 | 입력 브리프와 심사 기준을 고정한다. 최종 판단은 사용자에게 남긴다. |
| Team Chief | 팀 최고 책임자 | 팀의 철학, 북극성 목표, 최종 제안의 일관성을 책임진다. |
| Team Manager | 중간 관리자 | 작업을 역할별로 나누고 1년/12주/7일 산출물을 연결한다. |
| Architect | 설계자 | 목표 구조, 방향 논리, 시스템 모델을 만든다. |
| Operator | 구현자 | 12주 캠페인과 7일 첫 증거를 실행 가능한 계획으로 바꾼다. |
| Sentinel | 감시자 | 자기기만, 과소목표, 허황된 목표, 누락된 제약을 공격한다. |
| Synthesizer | 통합자 | 내부 비판을 반영해 팀 제안서를 정리한다. |

## 기본 경쟁 팀

### moonshot-team

목표를 의도적으로 크게 잡는다.

- Chief 질문: 지금 목표가 10배 커진다면 문제 정의가 어떻게 바뀌는가?
- Architect 초점: 레버리지, 플랫폼, 공개 산출물, 네트워크 효과
- Operator 초점: 큰 목표의 첫 관찰 가능한 증거
- Sentinel 공격: 허영, 도피성 거대 목표, 현재 자원과의 불연속

### compound-team

매주 쌓이는 구조를 설계한다.

- Chief 질문: 1년 동안 매주 쌓이면 압도적으로 달라지는 것은 무엇인가?
- Architect 초점: 반복 루프, 누적 자산, 학습 속도
- Operator 초점: 주간 리듬, 측정 지표, 회고 루틴
- Sentinel 공격: 안전한 작은 목표로 숨는 패턴, 지루함에 의한 이탈

### constraint-breaker-team

더 열심히 하는 대신 병목과 게임판을 바꾼다.

- Chief 질문: 지금 목표가 작은 이유가 능력 부족이 아니라 형식의 한계라면 무엇을 바꿔야 하는가?
- Architect 초점: 병목 지도, 대체 매체, 자동화, 팀/도구 재배치
- Operator 초점: 병목 하나를 제거하는 7일 실험
- Sentinel 공격: 도구 놀이, 회피성 시스템 구축, 실행 없는 구조 변경

### craft-team

오래 남는 실력과 작품을 중심에 둔다.

- Chief 질문: 이 목표가 끝났을 때 남는 실력과 작품은 무엇인가?
- Architect 초점: 기술 수련, 품질 기준, 대표작, 피드백 소스
- Operator 초점: 훈련 과제, 피드백 받는 장면, 공개/검수 루프
- Sentinel 공격: 완벽주의, 비공개 연습에 갇힘, 품질 명분의 지연

### operator-team

현실의 마찰과 실행 가능성을 중시한다.

- Chief 질문: 이번 주 안에 무엇이 관찰 가능해야 하는가?
- Architect 초점: 실행 순서, 리스크, 의존성, 의사결정 지점
- Operator 초점: 오늘/이번 주 행동, 산출물 파일, 검증 방법
- Sentinel 공격: 너무 안전한 일정, 큰 방향의 축소, 측정 불가능한 실행

## 팀 수 선택

- 감정적으로 무겁거나 상담 표면이면 `compound`, `constraint-breaker`, `craft` 3팀으로 시작한다.
- 전략/커리어/제품 방향이면 `moonshot`, `compound`, `constraint-breaker`, `operator` 4팀을 기본값으로 둔다.
- 큰 인생 방향, 사업, 장기 연구, 소설/작품 계획, 디자인 시스템처럼 선택 비용이 크면 5팀 모두 사용한다.

## 내부 비판 규칙

각 팀은 최종 제안 전에 Sentinel 라운드를 반드시 수행한다.

1. 과소목표인지 검사한다.
2. 허황된 목표인지 검사한다.
3. 사용자의 실제 제약을 무시했는지 검사한다.
4. 실행 증거가 7일 안에 나오는지 검사한다.
5. Team Chief가 비판을 반영해 제안서를 한 번 수정한다.

## 교차 비판 규칙

- 각 팀은 다른 팀에서 훔칠 만한 강점 1개를 명시한다.
- 각 팀은 다른 팀의 치명적 약점 1개를 명시한다.
- 최종 Judge Packet에는 원안과 비판 후 수정안을 구분한다.
