# TDD Workflow Guardrails

## 적용 대상

- 새 기능
- 버그 수정
- 동작 변경
- 리팩터링

문서, 일회용 탐색, 생성 산출물, 설정 파일은 예외가 될 수 있다. 예외가 필요하면 먼저 사용자에게 말한다.

## Red

- 원하는 외부 동작을 테스트 이름으로 쓴다.
- 하나의 테스트는 하나의 이유로 실패해야 한다.
- 테스트가 실패하는 것을 먼저 확인한다.
- 실패 원인이 오타, import 오류, fixture 오류가 아니라 기능 부재인지 확인한다.

## Green

- 테스트를 통과시키는 가장 작은 변경만 한다.
- 같은 단계에서 리팩터링과 기능 확장을 섞지 않는다.
- 테스트가 실패하면 프로덕션 코드를 고친다. 테스트 기대값을 현실에 맞춰 낮추지 않는다.

## Refactor

- Green 이후에만 이름, 중복, 경계, 모듈 구조를 정리한다.
- 리팩터링 중 테스트는 계속 Green이어야 한다.
- 추상화는 실제 중복 지식이나 두 번째 사용처가 있을 때만 만든다.

## 중단 신호

- 테스트가 바로 통과했다.
- 실패 이유를 설명할 수 없다.
- 구현 후 테스트를 맞추고 있다.
- 매몰 비용 때문에 잘못된 구현을 유지하고 있다.
- mock이 실제 동작보다 더 많이 검증되고 있다.

이 경우 현재 변경을 멈추고 테스트 의도를 다시 작성한다.

## 완료 기준

- 실패를 본 테스트가 있다.
- Green을 확인했다.
- 관련 기존 테스트가 통과한다.
- 새 실패 경로가 있으면 회귀 테스트가 있다.
- 검증하지 못한 항목은 최종 보고에 남긴다.
