# Software Engineering Laws as Guardrails

이 문서는 소프트웨어 공학 법칙을 에이전트가 실제 작업에서 적용할 수 있는 행동 규칙으로 바꾼다.

## Planning

| 법칙 | 행동 규칙 |
|------|-----------|
| Brooks의 법칙 | 늦어진 작업에 병렬 에이전트를 더 붙이기 전에 범위 축소, 의존성 제거, 검증 단위 분리를 먼저 검토한다. |
| Parkinson의 법칙 | 열린 작업은 작은 완료 조건과 검증 명령으로 닫는다. |
| 90-90 법칙 | "거의 끝"이라고 판단하기 전에 엣지 케이스, 실패 경로, 문서/검증 누락을 확인한다. |
| Goodhart의 법칙 | 커버리지, 파일 수, TODO 수 같은 지표를 목표로 착각하지 않는다. 사용자 가치와 실패 검출력을 같이 본다. |

## Architecture

| 법칙 | 행동 규칙 |
|------|-----------|
| Hyrum의 법칙 | 관찰 가능한 동작은 모두 계약이다. API 응답, 에러 문구, 정렬, 타이밍을 바꾸면 호환성 영향을 점검한다. |
| Gall의 법칙 | 작동하는 작은 시스템을 먼저 만든 뒤 확장한다. 처음부터 프레임워크나 거대한 추상화로 시작하지 않는다. |
| 누수 추상화의 법칙 | ORM, HTTP client, queue, UI abstraction이 숨긴 하위 동작을 디버깅할 수 있는 탈출구를 남긴다. |
| Tesler의 법칙 | 복잡성을 없앴다고 말하기 전에 누가 그 복잡성을 부담하는지 적는다. |
| Second-System 효과 | 성공한 첫 버전을 전면 재설계할 때는 기능 팽창과 과도한 일반화를 의심한다. |
| 분산 컴퓨팅의 오류 | 네트워크, 시간, 대역폭, 보안, 토폴로지, 운영 주체가 안정적이라는 가정을 금지한다. |

## Quality

| 법칙 | 행동 규칙 |
|------|-----------|
| Boy Scout 규칙 | 만진 파일의 작은 결함은 범위 안에서 개선한다. 대규모 리팩터링은 별도 작업으로 분리한다. |
| Murphy의 법칙 | 실패할 수 있는 입력, 외부 호출, 파일, 네트워크, 시간 의존성에 방어 경로를 둔다. |
| Postel의 법칙 | 입력 파싱은 현실적인 변형을 수용하되, 출력과 저장 계약은 엄격하게 유지한다. |
| 깨진 유리창 이론 | 깨진 테스트, 무시된 경고, 죽은 코드, 임시 우회를 방치하지 않는다. |
| 기술 부채 | 의도적 부채는 허용하지만 만료 조건, 후속 작업, 검증 위험을 함께 남긴다. |
| Kernighan의 법칙 | 디버깅하기 어려운 영리한 코드를 거부한다. 이름, 흐름, 실패 메시지가 읽혀야 한다. |
| Testing Pyramid | 빠른 단위 테스트를 기본으로 두고, 계약 경계에는 통합 테스트, 핵심 사용자 흐름에는 E2E를 둔다. |
| Pesticide Paradox | 기존 테스트가 못 잡은 버그에는 새 케이스를 추가한다. 같은 테스트 반복만으로 안심하지 않는다. |

## Design

| 법칙 | 행동 규칙 |
|------|-----------|
| DRY | 지식의 중복을 줄인다. 단, 우연히 비슷한 코드까지 성급하게 합치지 않는다. |
| KISS | 이해, 테스트, 디버깅이 쉬운 구조를 우선한다. |
| SOLID | 원칙 이름을 근거로 추상화를 늘리지 않는다. 변경 이유와 호출자를 기준으로 적용한다. |
| 디미터 법칙 | 깊은 체이닝과 내부 구조 노출을 줄이고, 경계 메서드를 만든다. |
| 최소 놀라움의 원칙 | 이름, UI, API가 예측 가능한 동작을 해야 한다. 반전 동작은 명시한다. |
| YAGNI | 현재 요구에 없는 기능은 만들지 않는다. 테스트가 있으면 나중에 추가해도 된다. |

## Decisions

| 법칙 | 행동 규칙 |
|------|-----------|
| Dunning-Kruger 효과 | 모르는 영역일수록 단정하지 말고 공식 문서, 코드, 테스트로 확인한다. |
| Hanlon의 면도날 | 이전 구현을 악의로 해석하지 않는다. 제약, 시간, 정보 부족을 먼저 고려한다. |
| Occam의 면도날 | 디버깅은 단순한 원인부터 확인한다. 복잡한 아키텍처 가설은 증거가 있을 때만 세운다. |
| 매몰 비용 오류 | 이미 작성한 코드라도 잘못된 방향이면 버린다. 미래 비용을 기준으로 판단한다. |
| 지도는 영토가 아니다 | 문서와 다이어그램보다 실제 코드, 런타임 동작, DB 상태를 우선 확인한다. |
| 확증 편향 | 선호 스택과 반대되는 증거도 찾는다. |
| Lindy 효과 | 새 도구보다 오래 검증된 도구가 더 나은 기본값일 때가 많다. |
| 제1원리 사고 | 관행을 따르기 전에 문제의 필수 조건을 분해한다. |
| 역전 사고 | 성공 경로만 보지 말고 실패시키는 방법을 먼저 나열한다. |
| 파레토 원칙 | 영향 큰 20%를 먼저 찾아 고친다. 단, 작은 결함 방치를 정당화하지 않는다. |
