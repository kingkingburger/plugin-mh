# Systematic Debugging: 체계적 디버깅

랜덤 수정은 시간 낭비이고 새로운 버그를 만든다. 빠른 패치는 근본 문제를 가린다.

> **핵심 원칙**: 근본 원인을 찾기 전에 수정을 시도하지 않는다. 증상 수정은 실패다.

---

## 절대 법칙

```
근본 원인 조사 없이 수정을 시도하지 않는다.
```

Phase 1을 완료하지 않았다면, 수정을 제안할 수 없다.

## 적용 대상

모든 기술적 문제에 사용:
- 테스트 실패
- 프로덕션 버그
- 예상치 못한 동작
- 성능 문제
- 빌드 실패
- 통합 이슈

**특히 이럴 때 사용:**
- 시간 압박을 받고 있을 때 (긴급 상황에서 추측이 유혹적)
- "빠른 수정 하나면 될 것 같을 때"
- 이미 여러 번 수정을 시도했을 때
- 이전 수정이 안 먹혔을 때

## 4단계 프로세스

각 단계를 완료해야 다음으로 진행할 수 있다.

### Phase 1: 근본 원인 조사

**수정을 시도하기 전에:**

**1. 에러 메시지를 꼼꼼히 읽는다**
- 에러나 경고를 건너뛰지 않는다
- 스택 트레이스를 끝까지 읽는다
- 라인 넘버, 파일 경로, 에러 코드를 메모한다
- 에러 메시지에 해결책이 그대로 들어 있는 경우가 많다

**2. 일관되게 재현한다**
- 신뢰할 수 있게 트리거할 수 있는가?
- 정확한 재현 단계는 무엇인가?
- 매번 발생하는가?
- 재현 불가 → 더 많은 데이터를 수집한다, 추측하지 않는다

**3. 최근 변경을 확인한다**
- 이 문제를 유발할 수 있는 변경이 무엇인가?
- git diff, 최근 커밋 확인
- 새로운 의존성, 설정 변경
- 환경 차이

**4. 멀티 컴포넌트 시스템에서 증거를 수집한다**

시스템이 여러 컴포넌트로 구성되어 있을 때 (API → 서비스 → DB, CI → 빌드 → 배포):

```
각 컴포넌트 경계마다:
  - 컴포넌트에 들어가는 데이터를 로그한다
  - 컴포넌트에서 나오는 데이터를 로그한다
  - 환경/설정 전파를 확인한다
  - 각 레이어의 상태를 점검한다

한 번 실행하여 어디에서 깨지는지 증거를 수집한다
→ 증거를 분석하여 실패하는 컴포넌트를 식별한다
→ 그 특정 컴포넌트를 조사한다
```

**5. 데이터 흐름을 추적한다**

에러가 콜스택 깊은 곳에 있을 때:
- 잘못된 값이 어디서 시작되는가?
- 누가 잘못된 값으로 이 함수를 호출했는가?
- 소스를 찾을 때까지 계속 역추적한다
- **증상이 아니라 소스를 수정한다**

### Phase 2: 패턴 분석

**수정 전에 패턴을 찾는다:**

1. **작동하는 예시를 찾는다**
   - 같은 코드베이스에서 유사하게 작동하는 코드를 찾는다
   - 깨진 것과 비슷한데 작동하는 것은 무엇인가?

2. **참조와 비교한다**
   - 패턴을 구현 중이라면 참조 구현을 **완전히** 읽는다
   - 훑어보지 않는다 — 모든 줄을 읽는다

3. **차이를 식별한다**
   - 작동하는 것과 깨진 것의 차이는 무엇인가?
   - 아무리 작아도 모든 차이를 나열한다
   - "그건 상관없을 거야"라고 가정하지 않는다

4. **의존성을 파악한다**
   - 이것이 필요로 하는 다른 컴포넌트는?
   - 어떤 설정, 환경이 필요한가?
   - 어떤 가정을 하고 있는가?

### Phase 3: 가설과 검증

**과학적 방법:**

1. **단일 가설을 세운다**
   - 명확하게 진술: "X가 근본 원인이라고 생각한다, 왜냐하면 Y이므로"
   - 적어둔다
   - 구체적으로, 모호하지 않게

2. **최소한으로 테스트한다**
   - 가설을 검증할 **가장 작은** 변경을 만든다
   - 한 번에 하나의 변수만
   - 여러 가지를 동시에 고치지 않는다

3. **계속하기 전에 확인한다**
   - 맞았다면 → Phase 4로
   - 틀렸다면 → 새로운 가설을 세운다
   - 위에 더 많은 수정을 얹지 **않는다**

4. **모르겠을 때**
   - "X를 이해하지 못한다"고 말한다
   - 아는 척하지 않는다
   - 도움을 요청한다

### Phase 4: 구현

**증상이 아니라 근본 원인을 수정한다:**

1. **실패하는 테스트 케이스를 만든다**
   - 가장 단순한 재현
   - 가능하면 자동화된 테스트
   - 수정 전에 반드시 있어야 한다

2. **단일 수정을 구현한다**
   - 식별된 근본 원인을 해결한다
   - 한 번에 **하나의** 변경
   - "겸사겸사" 개선 금지
   - 리팩토링 끼워넣기 금지

3. **수정을 검증한다**
   - 테스트가 이제 통과하는가?
   - 다른 테스트가 깨지지 않았는가?
   - 문제가 실제로 해결되었는가?

4. **수정이 안 먹히면**
   - **멈춘다**
   - 세어본다: 몇 번째 수정 시도인가?
   - 3회 미만: Phase 1로 돌아가서 새 정보로 재분석
   - **3회 이상: 멈추고 아키텍처를 의심한다 (아래 5번)**

5. **3회 이상 수정 실패: 아키텍처 의심**

   아키텍처 문제의 징후:
   - 각 수정이 다른 곳에서 새로운 공유 상태/결합/문제를 드러낸다
   - 수정에 "대규모 리팩토링"이 필요하다
   - 각 수정이 다른 곳에서 새로운 증상을 만든다

   **멈추고 근본적으로 질문한다:**
   - 이 패턴 자체가 근본적으로 건전한가?
   - "관성으로 고수하고 있는" 것은 아닌가?
   - 증상 수정 vs 아키텍처 리팩토링?

   **사용자와 논의한 후에 추가 수정을 시도한다.**

## Red Flags — 멈추고 프로세스를 따라라

이런 생각이 들면:
- "일단 빨리 고치고 나중에 조사하자"
- "X를 바꿔보고 되는지 보자"
- "여러 변경을 넣고 테스트 돌리자"
- "테스트 건너뛰고 수동으로 확인하지"
- "아마 X일 거야, 고쳐보자"
- "완전히 이해는 못 했지만 이게 될 수도"
- "한 번만 더 고쳐보자" (이미 2번 이상 시도)
- 조사 없이 해결책을 제안하고 있다

**이 모든 것의 답: STOP. Phase 1로 돌아간다.**

## 합리화 방지

| 핑계 | 현실 |
|------|------|
| "간단한 문제라 프로세스 필요 없어" | 간단한 문제에도 근본 원인이 있다. 프로세스는 간단한 버그에 빠르다. |
| "긴급이라 프로세스 따를 시간 없어" | 체계적 디버깅이 추측-시도 삽질보다 **빠르다**. |
| "일단 이거 먼저 해보고 조사하자" | 첫 수정이 패턴을 만든다. 처음부터 제대로 하라. |
| "수정 확인 후에 테스트 쓸게" | 테스트 안 된 수정은 지속되지 않는다. |
| "여러 수정 동시에 하면 시간 절약" | 뭐가 효과 있었는지 분리 불가. 새 버그 유발. |
| "참조가 너무 길어서 패턴만 적용할게" | 부분적 이해는 버그를 보장한다. 완전히 읽어라. |
| "문제가 보인다, 고치겠다" | 증상을 보는 것 ≠ 근본 원인을 이해하는 것. |
| "한 번만 더" (2회 이상 실패 후) | 3회 이상 = 아키텍처 문제. 패턴을 의심하라. |

## 빠른 참조

| Phase | 핵심 활동 | 성공 기준 |
|-------|----------|----------|
| **1. 근본 원인** | 에러 읽기, 재현, 변경 확인, 증거 수집 | 무엇이 왜 깨졌는지 이해 |
| **2. 패턴** | 작동하는 예시 찾기, 비교 | 차이점 식별 |
| **3. 가설** | 이론 수립, 최소 검증 | 확인 또는 새 가설 |
| **4. 구현** | 테스트 작성, 수정, 검증 | 버그 해결, 테스트 통과 |

## "근본 원인 없음"이 드러날 때

체계적 조사 결과 환경, 타이밍, 외부 요인 문제인 경우:

1. 프로세스를 완료한 것이다
2. 조사한 내용을 문서화한다
3. 적절한 처리를 구현한다 (재시도, 타임아웃, 에러 메시지)
4. 향후 조사를 위한 모니터링/로깅을 추가한다

**단:** 95%의 "근본 원인 없음"은 불완전한 조사다.

## 실전 효과

| | 체계적 접근 | 랜덤 수정 |
|--|-----------|----------|
| 수정 시간 | 15-30분 | 2-3시간 삽질 |
| 첫 시도 수정율 | ~95% | ~40% |
| 새 버그 유발 | 거의 없음 | 빈번 |

## 최종 규칙

```
수정 → 근본 원인 조사가 선행되어야 한다
그렇지 않으면 → 체계적 디버깅이 아니다
```

사용자의 명시적 허락 없이 프로세스를 건너뛰지 않는다.
