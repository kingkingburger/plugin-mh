# Ralph Prep: 랄프 전 철저한 준비

머릿속 아이디어 또는 정리된 문서를 받아, **유저 시나리오가 사진처럼 선명해질 때까지** 심층 인터뷰를 진행하고, 검증 방법까지 포함된 PRD 문서를 생성한다.

> **핵심 철학**: "애매모호함이라는 그림자가 거의 아예 남지 않을 정도로" 대화한다. 질문 수 제한 없음.

---

## 절대 원칙

1. **유저 시나리오 기준으로만 소통한다.** 개발/구현 이야기는 Phase 3 이전까지 철저히 제쳐둔다. AI가 기술 구현 세부사항으로 빠지려 하면 "지금은 유저 시나리오만 사진처럼 선명하게 싱크를 맞추고 싶어"라고 되돌린다.
2. **질문 수 제한 없음.** 20턴이든 30턴이든, 모호함이 완전히 사라질 때까지 계속한다. 단, Phase 전환 시점은 명확히 한다.
3. **모든 질문은 AskUserQuestion 도구로.** 일반 텍스트로 질문하지 않는다.
4. **가설 기반 질문.** 열린 질문 대신 구체적 옵션을 제시한다.
5. **아리까리한 결정은 그 자리에서 결론.** "나중에"로 넘기지 않는다.
6. **컨텍스트 보존.** 각 Phase 완료 시 중간 요약을 PRD 문서에 저장하여 컨텍스트 포화에 대비한다.

---

## 진입 판단

사용자의 입력 상태를 자동으로 판단한다:

| 입력 상태 | 판단 기준 | 시작 지점 |
|-----------|-----------|-----------|
| **아이디어만** | 한 줄~몇 줄 설명, 구체적 시나리오 없음 | Phase 1 처음부터 |
| **정리된 문서** | 파일 경로 제공 또는 구조화된 설명 | 문서를 읽고 분석 후, 부족한 Phase부터 |

정리된 문서가 제공된 경우:
1. Glob/Read로 문서를 읽는다
2. 어떤 Phase까지 커버되어 있는지 진단한다
3. 부족한 부분부터 인터뷰를 시작한다

### 기술 익숙도 진단

Phase 1 시작 전, 사용자의 기술 익숙도를 AskUserQuestion으로 확인한다:

```
questions:
  - question: "이 프로젝트에서 사용할 기술 스택에 대한 익숙도는 어느 정도인가요?"
    header: "기술 익숙도"
    options:
      - label: "잘 안다"
        description: "아키텍처 수준의 판단이 가능하고, LLM의 잘못된 선택을 잡아낼 수 있다"
      - label: "보통이다"
        description: "기본은 알지만, 세부 패턴이나 베스트 프랙티스는 확신이 없다"
      - label: "잘 모른다"
        description: "처음 쓰는 기술이거나, 경험이 거의 없다"
    multiSelect: false
```

이 결과에 따라 Phase 3 개발 계획의 상세도를 조정한다:
- **잘 안다**: 태스크 수준 계획으로 충분. 개발자 에이전트에 재량을 줄 수 있음
- **보통이다**: 파일/함수 수준 계획 필수. 주요 설계 결정을 계획에 명시
- **잘 모른다**: 파일/함수 수준 계획 + 각 결정의 근거를 상세히 기록. Phase 2에서 기술 관련 질문을 더 적극적으로. PRD 완성 후 `/expert-review`로 전문가 리뷰 권장

## Phase 1: 아이디에이션 & 주요 방향 의사결정

### 목적
기능의 개괄적인 사용 패턴을 잡는다. 디테일은 비워둔 채로 굵은 방향성을 결정한다.

### 프로토콜

1. **사용자의 아이디어를 그대로 기록**한다
2. **핵심 유저 시나리오를 구술**한다: "이 제품/기능을 사용하는 사람은 누구이고, 어떤 상황에서, 무엇을 하는가?"
3. **A/B 의사결정이 필요한 지점을 식별**한다
4. 의사결정이 필요하면 AskUserQuestion으로 옵션을 제시한다

### A/B 의사결정 시

사용자가 결정하기 어려워하면:

> **"이 결정이 어렵다면, 여러 관점에서 토론을 통해 결론을 내릴 수 있습니다. `/agent-arena`를 사용하여 찬반 토론을 진행해보세요."**

### Phase 1 완료 조건

- 핵심 유저 시나리오의 큰 그림이 합의됨
- 주요 방향성(A/B) 의사결정이 모두 완료됨
- **명시적 비목표(YAGNI)가 정의됨**
- 사용자가 "대략 이 방향이 맞다"고 확인

### 비목표(YAGNI) 정의

방향성 합의 후, 반드시 **하지 않을 것**을 명시적으로 정의한다. AskUserQuestion으로 확인:

> "이 프로젝트/기능에서 **의도적으로 하지 않을 것**은 무엇인가요? 예: 다국어 지원, 오프라인 모드, 관리자 대시보드 등"

비목표를 정하는 이유:
- 개발자 에이전트가 "있으면 좋겠다"고 판단해 범위를 넓히는 것을 방지
- 리뷰 시 비목표 관련 피드백을 걸러내는 기준이 됨
- 의사결정의 근거를 명확히 남김 (왜 안 하는지)

### Phase 1 완료 시 행동

중간 요약을 PRD 문서에 저장한다 (docs/prd/ 경로, 파일명은 `{날짜}-{주제슬러그}.md`):

```markdown
# {주제} PRD (작성 중)

## Phase 1: 아이디에이션 요약

### 핵심 유저 시나리오
{큰 그림 요약}

### 의사결정 기록
| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| ... | ... | ... |

### 명시적 비목표 (YAGNI)
| 비목표 | 제외 근거 |
|--------|-----------|
| ... | ... |

---
(Phase 2 이후 내용은 아래에 추가됨)
```

## Phase 2: 심층 인터뷰 — Unknown-Unknown 정의 (가장 중요)

### 목적
Phase 1에서 잡은 굵은 유저 시나리오를 **스텝 바이 스텝으로 잘게 쪼개어**, 모든 애매모호함을 제거한다.

### 프로토콜

1. **유저 시나리오를 화면 단위로 분해**한다
   - "사용자가 이 화면에서 무엇을 보는가?"
   - "어떤 버튼을 누르는가?"
   - "그 다음에 무슨 일이 일어나는가?"

2. **시각화** (사용자 선택에 따라)
   - 기본: 구술로 스텝 바이 스텝 설명
   - 옵션: **HTML 데모 화면을 직접 생성하여 브라우저에서 확인**

   시각화 요청 시 행동:
   1. 유저 시나리오의 각 화면을 HTML/CSS로 구성한다 (단일 HTML 파일)
   2. 실제 UI와 유사한 레이아웃, 버튼, 입력 필드를 포함한다
   3. 화면 간 전환 흐름을 탭/섹션으로 나누어 한 파일에서 확인 가능하게 한다
   4. `docs/prd/demos/{날짜}-{주제슬러그}-demo.html`에 저장한다
   5. Playwright의 `browser_navigate`로 파일을 열어 사용자에게 보여준다

   ```html
   <!-- 데모 화면 구성 예시 -->
   <div class="screen" id="screen-main">
     <h2>메인 화면</h2>
     <ul class="item-list">
       <li>항목 1</li>
       <li>항목 2</li>
     </ul>
     <button onclick="showScreen('detail')">+ 추가</button>
   </div>
   ```

   > **주의**: ASCII 코드로 화면을 설명하지 않는다. 반드시 HTML 데모 화면을 생성하여 실제 UI를 보여준다.

3. **모호한 부분을 모두 질문한다** — 질문 수 제한 없음
   - 예외 상황: "이 상태에서 뒤로가기를 누르면?"
   - 에러 케이스: "네트워크가 끊기면?"
   - 경계 조건: "데이터가 0개일 때? 1000개일 때?"
   - 권한: "로그인하지 않은 사용자가 접근하면?"
   - **보안/악용**: "악의적 사용자가 이 기능을 어떻게 악용할 수 있는가?"
     - 입력 조작: 특수문자, 인젝션, 경계값 넘기기
     - 권한 우회: 다른 사용자의 데이터에 접근 시도
     - 남용: 대량 요청, 스팸, 리소스 고갈
     - 예시: Stavros 사례에서 `"me@mydomain.com"@evildomain.com` 같은 이메일 주소로 와일드카드 허용목록을 우회하는 공격을 Phase 2에서 포착함

4. **새로운 의사결정이 발생하면 그 자리에서 결론**
   - 사용자에게 AskUserQuestion으로 옵션 제시
   - 결정하기 어려우면: "여러 관점에서 분석이 필요하시면 `/agent-arena`를 활용해보세요."
   - 사각지대가 의심되면: "숨겨진 가정을 점검하고 싶으시면 `/clarify unknown` 모드를 활용해보세요."
   - 관점 전환이 필요하면: "현재 접근 방식 자체를 재고하고 싶으시면 `/clarify metamedium` 모드를 활용해보세요."

5. **"개발 이야기 제쳐둠" 가드레일**
   - AI가 기술 구현으로 빠지려 하면 즉시 유저 시나리오로 되돌린다
   - "이건 어떻게 구현하느냐"가 아니라 "사용자가 어떤 경험을 하느냐"에 집중
   - 단, 사용자가 명시적으로 기술 논의를 요청하면 허용

### Phase 2 완료 조건

- 모든 유저 시나리오가 스텝 바이 스텝으로 분해됨
- 추가 질문이 더 이상 떠오르지 않음
- 사용자의 눈앞에 **제품이 어떻게 동작하는지 선명하게 그려짐**

### 완료 확인 방법

AskUserQuestion으로 확인:

```
questions:
  - question: "지금까지 정리된 유저 시나리오가 눈앞에 선명하게 그려지나요? 추가로 모호한 부분이 있나요?"
    header: "선명도 확인"
    options:
      - label: "선명하게 그려진다"
        description: "모든 시나리오가 명확하고, 추가 질문이 없다"
      - label: "아직 모호한 부분이 있다"
        description: "추가 질문이 필요한 부분이 남아있다"
      - label: "데모 화면으로 전체 흐름을 보여줘"
        description: "HTML 데모 화면을 생성하여 브라우저에서 확인하고 싶다"
    multiSelect: false
```

"아직 모호한 부분이 있다" → 추가 질문 계속
"데모 화면으로 보여줘" → HTML 데모 화면을 생성하고 브라우저에서 열어 확인한 후 다시 질문

### Phase 2 완료 시 행동

PRD 문서에 Phase 2 내용을 추가:

```markdown
## Phase 2: 유저 시나리오 상세

### 시나리오 1: {시나리오명}
**전제 조건**: {사용자 상태}
**단계**:
1. {스텝 1}
2. {스텝 2}
...

**예외 처리**:
- {예외 1}: {대응}
- {예외 2}: {대응}

### 시나리오 2: {시나리오명}
...

### 의사결정 기록 (Phase 2)
| 결정 사항 | 선택 | 근거 |
|-----------|------|------|
| ... | ... | ... |
```

## Phase 3: 검증 방법 설정 및 개발 계획

### 목적
만든 것이 성공적으로 동작하는지 검증할 수 있는 구체적인 방법을 설정하고, 개발 계획을 수립한다.

### 프로토콜

#### 3-1: 검증 도구 결정

AskUserQuestion으로 검증 환경을 확인:

```
questions:
  - question: "이 제품/기능을 검증할 때 어떤 도구/환경을 사용할까요?"
    header: "검증 도구"
    options:
      - label: "브라우저 자동화 (Playwright)"
        description: "웹 프로젝트 — 실제 브라우저에서 클릭/입력하며 E2E 검증"
      - label: "CLI 직접 실행"
        description: "CLI 도구 — 실제 명령어를 실행하고 출력을 확인"
      - label: "API 호출 (curl/httpie)"
        description: "API 서버 — 실제 엔드포인트를 호출하여 응답 검증"
    multiSelect: false
```

> **핵심 원칙**: "유저와 완전히 동일한 환경에서 검증." curl이 아닌 실제 제품을 사용하는 것과 동일한 방식으로 검증한다.

#### 3-2: 검증 시나리오 작성

Phase 2의 유저 시나리오를 기반으로 검증 시나리오를 생성:

- **정상 시나리오**: 유저 시나리오 각 스텝이 정상 동작
- **엣지 케이스**: 이상한 유저 패턴, 경계 조건, 에러 상황
- **회귀 테스트**: 기존 기능이 깨지지 않는지

빡세게 작성한다 — 나이브한 테스트만 만들지 않는다.

#### 3-3: 개발 계획 수립

이제 비로소 "어떻게 개발할 것인가"를 논의한다:

1. Phase 2의 유저 시나리오를 기반으로 개발 태스크를 도출
2. 사이즈가 크면 Phase N개로 분리
3. 각 Phase별 산출물과 완료 조건을 정의
4. **파일/함수 수준 저수준 계획 작성** (기술 익숙도가 "보통" 이상일 때 필수)

#### 3-4: 저수준 계획 인터뷰 (파일/함수 수준)

> **왜 이렇게까지 하는가**: Stavros 방법론의 핵심 — 계획이 저수준일수록 개발자 에이전트가 잘못된 선택을 내릴 여지가 줄어든다. "무엇을 어디에 넣을지"를 아키텍트가 결정하고, 개발자는 "어떻게 작성할지"만 담당한다.

##### 3-4-1: 코드베이스 탐색

먼저 기존 코드베이스를 Glob/Read/Grep으로 탐색하여 **현재 패턴을 파악**한다:
- 기존 프로젝트 구조와 디렉토리 컨벤션
- 유사한 기능이 이미 어떻게 구현되어 있는지
- 재사용 가능한 기존 모듈/함수
- Config 구조, 타입 정의 패턴, 에러 처리 패턴

탐색 결과를 사용자에게 공유하고, 기존 패턴을 따를지 새 패턴을 도입할지 AskUserQuestion으로 확인한다.

##### 3-4-2: 저수준 설계 질문

각 태스크에 대해 다음을 **사용자와 함께 결정**한다. 모든 질문은 AskUserQuestion으로, 가설 옵션을 제시한다:

1. **파일 배치**: "이 기능을 어디에 넣을까요?"
   - 기존 코드 패턴을 분석한 후 구체적 경로를 제안
   - 예: "`src/channels/email.ts`를 새로 만들까요, 아니면 기존 `src/channels/index.ts`에 추가할까요?"

2. **함수 경계**: "이 로직을 어떤 단위로 나눌까요?"
   - 함수 시그니처(이름, 파라미터, 반환값)를 제안
   - 예: "`parseInboundEmail(raw: string): ParsedEmail` 과 `sendOutboundEmail(to: string, body: string): void`로 분리하는 게 맞을까요?"

3. **의존성 선택**: "이 작업에 외부 패키지가 필요한가요?"
   - 후보를 제시하고 선택 근거를 함께 논의
   - 예: "MIME 파싱에 `mailparser`(가볍고 충분) vs `postal-mime`(WASM, 더 빠름) 중 어떤 걸 쓸까요?"

4. **Config 구조**: "설정은 어떤 형태로?"
   - 기존 Config 파일의 구조를 보여주고, 새 필드 위치/형태를 제안
   - 필수 vs 선택 필드, 기본값을 함께 결정

5. **에러 처리 전략**: "실패 시 어떻게?"
   - 재시도, 무시, 알림, 롤백 중 선택
   - 예: "이메일 발송 실패 시 — 3회 재시도 후 로그만 남길까요, 사용자에게 알릴까요?"

6. **기존 코드 재사용**: "이미 있는 것 중 쓸 수 있는 것은?"
   - 코드베이스 탐색에서 발견한 재사용 가능 모듈을 제시
   - LLM이 놓칠 수 있는 기존 유틸/헬퍼를 사용자가 짚어줄 수 있음
   - 예: "기존 `normalizeIdentifier()` 함수가 이메일에도 적용 가능한데, 이걸 확장할까요?"

7. **타입/인터페이스 설계**: "데이터 구조를 어떻게 잡을까요?"
   - 주요 타입/인터페이스의 필드 구성을 제안
   - 기존 타입과의 관계(확장, 합성, 독립)를 확인

##### 3-4-3: 맹점 확인

Stavros 방법론에서 가장 중요한 부분 — LLM이 코드베이스에서 특정 요소를 놓칠 수 있다:

> AskUserQuestion: "제가 탐색한 코드베이스에서 놓친 것이 있나요? 예를 들어, 반드시 사용해야 하는 기존 함수, 지켜야 하는 패턴, 또는 건드리면 안 되는 부분이 있다면 알려주세요."

사용자가 "Y를 사용해야 한다"고 지시하면 계획을 즉시 수정한다.

##### 3-4-4: 저수준 계획 확정

모든 질문이 해소되면, 최종 계획을 정리한다:

- **신규 파일**: 파일 경로, 역할, 주요 export/함수 목록
- **수정 파일**: 파일 경로, 수정할 함수/블록, 변경 내용 요약
- **삭제 파일**: 파일 경로, 삭제 근거
- **의존성**: 새로 추가할 패키지와 선택 근거

기술 익숙도에 따른 상세도:
- **잘 안다**: 태스크 + 수정 파일 목록이면 충분. 3-4-2 질문을 간소화 가능
- **보통이다**: 태스크 + 파일 + 함수 시그니처 + 주요 설계 결정 근거
- **잘 모른다**: 위 전부 + 각 결정의 대안과 제외 근거까지 기록. 3-4-2의 모든 항목을 빠짐없이 질문

### Phase 3 완료 조건

- 검증 도구/환경이 결정됨
- 정상 시나리오 + 엣지 케이스 검증 시나리오가 작성됨
- 개발 계획이 Phase별로 수립됨
- **저수준 계획 인터뷰 완료**: 파일 배치, 함수 경계, 의존성, Config, 에러 처리, 재사용, 타입 설계가 모두 결정됨
- **맹점 확인 완료**: 사용자에게 놓친 부분이 없는지 최종 확인함
- Definition of Done이 정의됨

## Anti-Patterns

| 하지 말 것 | 왜 |
|-----------|-----|
| Phase 2 전에 기술 구현을 논의 | 유저 시나리오가 흐려짐 |
| "나중에 결정하자"로 넘기기 | 모호함이 랄프까지 전파됨 |
| 열린 질문 ("어떻게 하고 싶어?") | 인지 부하. 가설 옵션을 제시하라 |
| 나이브한 검증 시나리오만 작성 | 실제 사용 시 온갖 버그 터짐 |
| 한 세션에서 모든 것을 기억하려 함 | Phase 완료 시 반드시 문서에 저장 |
| 질문 상한을 임의로 설정 | 모호함이 남으면 의미 없음 |
| 비목표를 정의하지 않음 | 개발자 에이전트가 범위를 임의로 넓힘 |
| 보안/악용 시나리오를 건너뜀 | 배포 후 발견하면 수정 비용이 기하급수적 |
| 태스크 수준만의 추상적 계획 | 개발자 에이전트에 재량이 커지면 잘못된 선택 누적 |

## 다른 스킬 활용 안내

ralph-prep 진행 중 아래 상황이 발생하면 해당 스킬을 안내한다:

| 상황 | 안내할 스킬 | 안내 문구 |
|------|-----------|-----------|
| A/B 의사결정이 어려움 | `/agent-arena` | "여러 관점에서 찬반 토론을 통해 결론을 내릴 수 있습니다." |
| 숨겨진 가정/사각지대 의심 | `/clarify unknown` | "Known/Unknown 4분면 분석으로 사각지대를 점검할 수 있습니다." |
| 접근 방식 자체를 재고 | `/clarify metamedium` | "Content vs Form 관점에서 더 높은 레버리지를 찾을 수 있습니다." |
| 목표가 너무 보수적 | `/moonshot` | "목표를 상향 조정하고 싶다면 moonshot 프레임워크를 활용해보세요." |
| 기술 스택 결정 필요 | `/tech-decision` | "기술 의사결정이 필요하면 체계적 분석을 진행할 수 있습니다." |
| PRD 완성 후 검증 필요 | `/expert-review` | "PRD를 다른 관점에서 리뷰받고 싶다면 전문가 리뷰를 활용해보세요." |
