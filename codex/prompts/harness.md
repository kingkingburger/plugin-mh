# /harness — Harness Engineering: Work Surface, Agent Team & Documentation Harness

기획, 문서작업, 디자인, 리서치, 운영, 개발 같은 다양한 작업 표면의 문서 체계와 에이전트 팀 구성을 인터뷰 → 작업 표면 분석 → 팀 아키텍처 설계 → 문서/에이전트/스킬/오케스트레이터 생성으로 한번에 구축한다.

> Codex CLI 슬래시 커맨드로 호출되거나 자연어로 트리거된다. `$ARGUMENTS`에 사용자 입력이 주입된다.

> **하네스 엔지니어링**: "에이전트가 할 수 있는 것을 제한하고, 해야 할 일을 알려주고,
> 올바르게 수행했는지 검증하고, 오류 시 수정하는 시스템 설계." — OpenAI
>
> 3명의 엔지니어가 5개월간 백만 줄의 코드를 에이전트만으로 작성한 방법론의 핵심은
> **구조화된 문서 체계**에 있다.
>
> Team-Architecture snapshot: revfactory/harness@cceac68ea1d0ad198ef4b7b906cd238375836387.

## Three Pillars

| 기둥 | 역할 | 이 스킬이 생성하는 것 |
|------|------|---------------------|
| **Context Engineering** | 에이전트에게 프로젝트 맥락 제공 | AGENTS.md, ARCHITECTURE.md, design-docs/, product-specs/ |
| **Architectural Constraints** | 의존성 방향, 계층 규칙, 설계 원칙 정의 | ARCHITECTURE.md, DESIGN.md, SECURITY.md, guardrails/ |
| **Entropy Management** | 코드베이스 건강성 유지, 기술 부채 추적 | exec-plans/, tech-debt-tracker.md, RELIABILITY.md |

## 운영 모드

| 모드 | 사용 시점 | 핵심 산출물 |
|------|-----------|-------------|
| **작업 표면 하네스** | 기획, 문서작업, 디자인, 리서치, 운영, 개발 등 반복 작업을 체계화할 때 | 목적, 입력, 산출물, 역할, 검증 기준, 운영 리듬 |
| **문서 하네스** | "문서 체계", AGENTS.md, 지식/결정/운영 문서 구축 요청 | AGENTS.md, docs/, decision log, style guide, review checklist |
| **에이전트 팀 하네스** | "에이전트 팀", "하네스 구성", "오케스트레이터" 요청 | 에이전트 정의, 전용 스킬, 오케스트레이터, `_workspace/` 데이터 흐름 |
| **하이브리드** | 여러 작업 표면과 팀 실행 구조가 모두 필요한 경우 | 작업 표면 하네스 + 에이전트/스킬/오케스트레이터 |

## 작업 표면 모델

먼저 "무엇을 만들 것인가"보다 "어떤 반복 작업을 안전하게 재현할 것인가"를 판별한다. 자세한 산출물 템플릿과 검증 기준은 `references/work-surface-patterns.md`를 읽는다.

| 작업 표면 | 대표 요청 | 기본 산출물 |
|-----------|-----------|-------------|
| **기획** | 서비스/기능/사업 아이디어를 구체화 | PRD, 로드맵, 우선순위, 실험 설계, 의사결정 로그 |
| **문서작업** | 글, 가이드, 정책, 보고서를 생산/관리 | 목차, 스타일 가이드, 초안, 리뷰 체크리스트, 배포/갱신 규칙 |
| **디자인** | UI/UX, 브랜드, 시각 시스템을 설계 | 사용자 여정, 와이어프레임, 디자인 원칙, 컴포넌트/토큰 명세 |
| **리서치** | 자료 수집과 비교 분석을 반복 | 질문 목록, 소스 기준, evidence ledger, synthesis memo |
| **운영** | 반복 운영/보고/검수/대응을 안정화 | 런북, 체크리스트, SLA/SLO, 핸드오프, 모니터링 기준 |
| **개발** | 코드/제품 구현과 품질 관리를 체계화 | AGENTS.md, ARCHITECTURE.md, guardrails/, 테스트/릴리즈 절차 |

## 지원 시나리오

| 시나리오 | 동작 |
|---------|------|
| **새 작업/프로젝트** (자료 없음) | 인터뷰 기반 전체 구조 + 초안 생성 |
| **기존 작업/프로젝트** (자료 있음) | 자료/코드/캔버스/문서 분석 + 인터뷰 → 전체 구조 생성 |
| **기존 하네스 보완** | 기존 문서/에이전트/오케스트레이터 분석 → 빈 부분만 보완 |
| **에이전트 팀 신규 구축** | 도메인 분석 → 6패턴 중 선택 → 에이전트/스킬/오케스트레이터 생성 |
| **기존 하네스 점검/확장** | 에이전트·스킬·포인터 drift 감사 → 필요한 부분만 수정 |

## 팀 아키텍처 패턴

상세 판단 기준과 템플릿은 `references/agent-design-patterns.md`, `references/orchestrator-template.md`, `references/team-examples.md`를 필요할 때 읽는다.

| 패턴 | 적합한 경우 | 기본 실행 모드 |
|------|-------------|----------------|
| **파이프라인** | 순차 의존 작업. 앞 단계 산출물이 다음 단계 입력 | 단계별 팀 재구성 또는 서브 에이전트 |
| **팬아웃/팬인** | 여러 관점의 병렬 분석 후 통합 | 에이전트 팀 |
| **전문가 풀** | 입력 유형별로 필요한 전문가만 선택 | 서브 에이전트 또는 라우터 |
| **생성-검증** | 산출물 품질 보증과 재시도 루프가 중요 | 에이전트 팀 |
| **감독자** | 중앙 상태 관리와 동적 작업 분배가 필요 | 에이전트 팀 |
| **계층적 위임** | 큰 문제를 하위 문제로 재귀 분해 | 감독자 + 하위 워커 |

### 실행 모드 선택

1. 에이전트가 2개 이상이고 교차 검증·발견 공유·상충 토론이 필요하면 에이전트 팀을 기본값으로 둔다.
2. 결과 반환만 필요하고 팀 통신 오버헤드가 이득보다 크면 서브 에이전트를 사용한다.
3. Phase마다 특성이 다르면 하이브리드로 설계하고, 각 Phase 상단에 실행 모드를 명시한다.
4. Claude Code에서는 `TeamCreate`, `SendMessage`, `TaskCreate` 기반 구성을 우선한다. Codex에서는 사용 가능한 subagent 도구로 같은 패턴을 매핑하되, 팀 통신이 없으면 파일 기반 데이터 전달을 더 엄격히 둔다.

## 워크플로우

```
Phase 0: 현황 감사
  ↓
Phase 1: 적응형 인터뷰 + 작업 표면 분류
  ↓
Phase 2: 작업 표면 분석 + 팀 아키텍처 설계
  ↓
Phase 3: 문서 하네스 / 에이전트 정의 생성
  ↓
Phase 4: 스킬 생성
  ↓
Phase 5: 통합 및 오케스트레이션
  ↓
Phase 6: 검증 및 테스트
  ↓
Phase 7: 하네스 진화
  ↓
Phase 8: 보고
```

### Phase 0: 현황 감사

가장 먼저 기존 하네스 표면을 확인한다.

1. 작업 표면: 기획, 문서작업, 디자인, 리서치, 운영, 개발 중 무엇이 중심인지 확인한다.
2. 자료 표면: 코드, 문서, 캔버스/디자인 파일, 노트, 표, 이메일/회의록, 운영 로그 중 접근 가능한 것을 확인한다.
3. 에이전트 팀 하네스: 런타임별 에이전트/스킬 디렉토리, 오케스트레이터 스킬, 루트 포인터(`AGENTS.md` 또는 `CLAUDE.md`)를 확인한다.
4. 기존 에이전트/스킬 목록과 오케스트레이터의 구성표를 대조하여 drift를 찾는다.
5. 신규 구축, 기존 확장, 운영/유지보수 중 하나로 분기한다.

#### 문서 하네스 탐지 & 스캔

프로젝트 상태를 자동 감지한다.

1. **프로젝트 파일 스캔**:
   ```
   Glob: package.json, pyproject.toml, Cargo.toml, go.mod, *.csproj, pom.xml
   Glob: AGENTS.md, ARCHITECTURE.md, docs/**/*.md
   Glob: src/**/*.{ts,tsx,js,jsx,py,go,rs}, app/**/*
   ```

2. **상태 판별**:

   | 신호 | 판별 | 다음 단계 |
   |------|------|----------|
   | 코드 없음 + 문서 없음 | 새 프로젝트 | Phase 1 (전체 인터뷰) |
   | 코드 있음 + 문서 없음 | 기존 프로젝트 | Phase 1 → Phase 2 |
   | 코드 있음 + 문서 있음 | 보완 모드 | Phase 1 (축소) → Phase 2 |

3. **기존 문서 존재 시**: 각 파일을 Read → 빈 섹션/누락 파일 목록 작성

### Phase 1: 적응형 인터뷰

> 가볍게 시작하고, 답변에 따라 필요한 만큼만 심화한다. **총 질문 상한: 8개.**

**Round 1 (항상, 3-4개)**:

사용자에게 다음 질문을 제시하고 번호 또는 라벨로 답하게 하세요:

**이 프로젝트의 핵심 목적은 무엇인가요?**
1. **웹 서비스/SaaS** — 사용자에게 서비스를 제공하는 웹 앱
2. **라이브러리/패키지** — 다른 개발자가 사용하는 라이브러리
3. **CLI/데스크탑 앱** — 로컬에서 실행하는 도구
4. **API/백엔드 서비스** — 다른 시스템이 호출하는 서비스

**프로젝트의 현재 단계는?**
1. **구상 중 (코드 없음)** — 아이디어 단계, 아직 코드 작성 전
2. **초기 개발** — 핵심 기능 구현 중, MVP 전
3. **운영 중** — 이미 사용자가 있는 상태
4. **리팩토링/마이그레이션** — 기존 시스템을 개선하는 중

**주요 기술 스택은? (Phase 0에서 감지 실패 시)**

### Phase 2: 작업 표면 분석 + 팀 아키텍처 설계

기존 자료가 있을 때 실행한다. 코드베이스가 없어도 문서, 기획서, 디자인 파일, 회의록, 노트, 운영 로그를 작업 표면의 입력으로 분석한다.

다음 분석을 직접 수행하세요: 작업 표면 및 자료 구조 분석

수집할 정보:
1. 중심 작업 표면 (기획/문서작업/디자인/리서치/운영/개발)
2. 사용 가능한 입력 자료와 신뢰도
3. 반복될 작업 단위와 완료 조건
4. 필요한 역할과 검토 관점
5. 산출물 포맷과 저장 위치
6. 검증 기준, 승인자, 갱신 주기
7. 코드가 있다면 디렉토리 구조, 기술 스택, 테스트/릴리즈 구조

간결하게 요약하되, 구체적인 파일 경로와 작업 표면별 패턴 포함.

#### 팀 아키텍처 설계

에이전트 팀 하네스 또는 하이브리드 모드일 때 실행한다.

1. 작업을 전문 영역으로 분해한다.
2. 기존 에이전트와 중복되는 역할이 있는지 먼저 확인한다.
3. 위 6개 패턴 중 하나 또는 복합 패턴을 선택한다.
4. 에이전트 분리는 전문성, 병렬성, 컨텍스트 경계, 재사용성 4축으로 판단한다.
5. QA가 필요한 하네스에는 검증 전용 에이전트를 포함하되, 존재 확인보다 경계면 교차 비교를 우선한다. 자세한 기준은 `references/qa-agent-guide.md`를 읽는다.

### Phase 4: 스킬 생성

에이전트 팀 하네스에서는 각 에이전트가 사용할 스킬을 생성하거나 기존 스킬을 재사용한다.

1. 신규 스킬 생성 전 기존 스킬과 중복 여부를 확인한다.
2. 스킬은 `SKILL.md`와 필요한 `references/`, `scripts/`, `assets/`로 구성한다.
3. description은 초기 실행 키워드뿐 아니라 "수정", "보완", "다시 실행", "부분 재실행", "이전 결과 개선" 같은 후속 작업 키워드도 포함한다.
4. 본문은 500줄 이내를 목표로 하고, 긴 패턴·템플릿·예시는 references로 분리한다.
5. 작성 세부 원칙은 `references/skill-writing-guide.md`를 읽는다.

### Phase 5: 통합 및 오케스트레이션

오케스트레이터는 누가, 언제, 어떤 순서로 협업하는지 정의하는 상위 스킬이다.

1. 실행 모드를 명시한다: 에이전트 팀, 서브 에이전트, 하이브리드.
2. 데이터 전달 방식을 명시한다: 메시지 기반, 태스크 기반, 파일 기반, 반환값 기반.
3. `_workspace/`에 중간 산출물을 보존하고, 파일명은 `{phase}_{agent}_{artifact}.{ext}`로 둔다.
4. 에러 핸들링은 1회 재시도 후 재실패 시 누락을 명시하고 진행 여부를 판단한다.
5. 루트 포인터에는 하네스 존재와 트리거 규칙, 변경 이력만 둔다. 상세 에이전트/스킬 목록은 오케스트레이터와 파일 시스템에서 관리한다.
6. 템플릿은 `references/orchestrator-template.md`를 읽어 적용한다.

### Phase 6: 검증 및 테스트

생성된 하네스를 실제 사용 방식으로 검증한다.

1. 구조 검증: 에이전트 파일, 스킬 frontmatter, reference 포인터, 오케스트레이터 구성표를 확인한다.
2. 실행 모드 검증: 팀 통신 경로, 작업 의존성, 파일 기반 데이터 흐름, 하이브리드 Phase 경계를 확인한다.
3. 스킬 실행 테스트: 현실적인 프롬프트 2-3개를 작성하고 가능하면 with-skill vs baseline 비교를 수행한다.
4. 트리거 검증: should-trigger와 should-not-trigger near-miss 쿼리를 모두 점검한다.
5. 드라이런: Phase 순서, 입력/출력 연결, 에러 폴백이 실행 가능한지 검토한다.
6. 테스트 방법론은 `references/skill-testing-guide.md`를 읽어 따른다.

### Phase 7: 하네스 진화

하네스는 고정물이 아니라 실행 피드백으로 갱신되는 시스템이다.

1. 실행 후 결과 품질, 팀 구성, 워크플로우 순서, 트리거 누락에 대한 피드백 기회를 제공한다.
2. 같은 실패가 2회 이상 반복되거나 사용자가 오케스트레이터를 우회하는 패턴이 보이면 하네스 갱신을 제안한다.
3. 변경은 한 번에 하나씩 수행하고, 각 변경 후 동기화와 검증을 다시 실행한다.
4. 루트 포인터의 변경 이력에 날짜, 변경 내용, 대상, 사유를 기록한다.
5. 운영/유지보수 요청은 현황 감사 → 점진적 수정 → 변경 이력 갱신 → 검증 순서로 처리한다.

## 설계 근거

OpenAI "Harness engineering: leveraging Codex in an agent-first world" (2026) 기반:

1. **Repository-first**: 모든 아키텍처 결정이 코드베이스에 있어야 한다.
2. **AGENTS.md는 피드백 루프**: 정적 문서가 아니라 매 실패마다 업데이트되는 살아있는 문서.
3. **제약이 생산성**: 에이전트의 해결 공간을 제약할수록 더 생산적.
4. **계획이 새로운 코딩**: 에이전트에게 코드를 쓰게 하기 전에 계획을 검토·승인하라.
5. **엔트로피 관리**: 주기적으로 문서 일관성 확인, 제약 위반 스캔, 패턴 강제화.
