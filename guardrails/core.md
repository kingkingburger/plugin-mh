# Core Guardrails

## 작업 방식

- 먼저 현재 코드와 문서를 읽고, 기존 패턴을 따른다.
- 구현 세부가 열려 있으면 보수적인 선택을 한다.
- 코드 변경은 요청 범위에 가깝게 유지한다.
- 사용자가 만든 변경을 되돌리지 않는다.
- 새 추상화는 실제 두 번째 사용처가 생기기 전까지 만들지 않는다.
- "나중에 유연하게"라는 이유만으로 옵션, props, 공용 유틸을 추가하지 않는다.

## 검증 기준

- 파일 존재만으로 완료를 보고하지 않는다.
- 동작 변경에는 집중 테스트를 실행한다.
- 버그 수정에는 실패를 재현하는 테스트나 검증 절차를 남긴다.
- Windows 작업에서도 PowerShell과 Git Bash를 모두 주요 셸로 취급한다.
- 셸 차이가 영향을 줄 수 있는 스크립트는 `.ps1`과 `.sh` 경로를 각각 검증한다.
- 검증하지 못한 경우에는 이유와 남은 위험을 명시한다.

## 설계 기준

- 복잡성은 제거되지 않고 이동한다. 사용자가 감당할 복잡성과 시스템 내부가 감당할 복잡성을 구분한다.
- API, CLI, 파일 포맷, 에러 메시지, 정렬 순서는 관찰 가능한 계약으로 취급한다.
- 빠른 길을 선택할 수는 있지만, 의도적 부채라면 상환 조건을 적는다.
- 성능 판단은 측정 가능한 병목이나 명확한 데이터 접근 패턴을 기준으로 한다.

## 문서 기준

- 에이전트용 문서는 실행 가능한 규칙과 명령을 우선한다.
- 제품/정책 사실을 저장소에서 확인할 수 없으면 `[TBD: ...]`로 표시한다.
- 오래된 계획 문서보다 현재 파일 시스템과 테스트 결과를 우선한다.
