# TDD: Test-Driven Development

테스트를 먼저 쓴다. 실패를 확인한다. 최소한의 코드로 통과시킨다.

> **핵심 원칙**: 테스트가 실패하는 걸 보지 않았다면, 그 테스트가 올바른 것을 검증하는지 알 수 없다.

---

## 절대 법칙

```
실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다.
```

테스트 전에 코드를 썼다면? **삭제하고 다시 시작한다.**

- "참고용"으로 남기지 않는다
- "적응"시키면서 테스트를 쓰지 않는다
- 보지도 않는다
- 삭제는 삭제다

## 적용 대상

**항상 적용:**
- 새 기능 구현
- 버그 수정
- 리팩토링
- 동작 변경

**예외 (사용자 확인 필요):**
- 일회용 프로토타입
- 생성된 코드 (codegen)
- 설정 파일

## RED → GREEN → REFACTOR

### RED: 실패하는 테스트 작성

하나의 동작을 검증하는 최소한의 테스트를 작성한다.

**좋은 테스트:**
- 이름이 동작을 설명한다
- 하나의 것만 테스트한다
- 실제 코드를 사용한다 (mock은 불가피할 때만)

**나쁜 테스트:**
- `test('works')` — 이름이 모호
- 이름에 "and"가 있다 → 분리해야 한다
- mock을 테스트하지 실제 코드를 테스트하지 않는다

### RED 검증: 실패를 확인한다

**필수. 절대 건너뛰지 않는다.**

```bash
# 테스트 실행
npm test path/to/test.test.ts  # 또는 프로젝트의 테스트 명령
```

확인 사항:
- 테스트가 실패한다 (에러가 아니라 실패)
- 실패 메시지가 예상한 것과 같다
- 기능이 없어서 실패한다 (오타가 아니라)

**테스트가 통과한다면?** 이미 존재하는 동작을 테스트하고 있다. 테스트를 수정해라.

### GREEN: 최소한의 코드

테스트를 통과시키는 **가장 단순한** 코드를 작성한다.

- 기능을 추가하지 않는다
- 다른 코드를 리팩토링하지 않는다
- 테스트 범위를 넘어서 "개선"하지 않는다

### GREEN 검증: 통과를 확인한다

**필수.**

```bash
npm test path/to/test.test.ts
```

확인 사항:
- 테스트가 통과한다
- 다른 테스트도 여전히 통과한다
- 경고나 에러 없이 깨끗하다

**테스트 실패?** 코드를 수정한다, 테스트를 수정하지 않는다.
**다른 테스트 실패?** 지금 바로 수정한다.

### REFACTOR: 정리

GREEN 이후에만:
- 중복 제거
- 이름 개선
- 헬퍼 추출

테스트를 GREEN으로 유지한다. 동작을 추가하지 않는다.

### 반복

다음 기능을 위한 다음 실패하는 테스트로.

## 버그 수정 예시

**버그:** 빈 이메일이 허용됨

**RED**
```typescript
test('빈 이메일을 거부한다', async () => {
  const result = await submitForm({ email: '' });
  expect(result.error).toBe('이메일은 필수입니다');
});
```

**RED 검증**
```bash
$ npm test
FAIL: expected '이메일은 필수입니다', got undefined
```

**GREEN**
```typescript
function submitForm(data: FormData) {
  if (!data.email?.trim()) {
    return { error: '이메일은 필수입니다' };
  }
  // ...
}
```

**GREEN 검증**
```bash
$ npm test
PASS
```

**REFACTOR** — 필요 시 다른 필드에도 적용할 수 있도록 검증 추출.

## 합리화 방지

| 핑계 | 현실 |
|------|------|
| "너무 간단해서 테스트 필요 없어" | 간단한 코드도 깨진다. 테스트는 30초면 된다. |
| "나중에 테스트 추가할게" | 이미 통과하는 테스트는 아무것도 증명하지 못한다. |
| "이미 수동 테스트 했어" | 수동은 기록이 없고 재실행 불가. 체계적이지 않다. |
| "X시간 작업한 걸 삭제하라고?" | 매몰 비용 오류. 검증 안 된 코드를 유지하는 게 낭비다. |
| "참고용으로 남겨두고 테스트부터 쓸게" | 결국 적응시킨다. 그건 테스트-후 작성이다. 삭제해라. |
| "먼저 탐색이 필요해" | 좋다. 탐색을 끝내고 버려라. 그 다음 TDD로 시작. |
| "TDD가 나를 느리게 해" | TDD가 디버깅보다 빠르다. |
| "기존 코드에 테스트가 없는데" | 지금 개선하고 있다. 건드리는 코드에 테스트를 추가해라. |
| "이건 실용적 판단이야" | TDD가 실용적이다. "실용적" 지름길 = 프로덕션에서 디버깅 = 더 느림. |

## Red Flags — 멈추고 다시 시작

이런 생각이 들면 **STOP**:

- 테스트 전에 코드를 작성했다
- 구현 후에 테스트를 추가했다
- 테스트가 바로 통과했다
- 왜 실패했는지 설명할 수 없다
- "이번 한 번만" 합리화하고 있다
- "참고용으로 남겨두자"
- "X시간 작업했으니 삭제는 낭비"

**이 모든 것의 답: 코드 삭제. TDD로 다시 시작.**

## 완료 전 체크리스트

- [ ] 모든 새 함수/메서드에 테스트가 있다
- [ ] 각 테스트가 실패하는 것을 확인했다
- [ ] 각 테스트가 예상된 이유로 실패했다 (기능 부재, 오타 아님)
- [ ] 각 테스트를 통과시키는 최소한의 코드를 작성했다
- [ ] 모든 테스트가 통과한다
- [ ] 출력이 깨끗하다 (에러, 경고 없음)
- [ ] 테스트가 실제 코드를 사용한다 (mock은 불가피한 경우만)
- [ ] 엣지 케이스와 에러 경우를 커버했다

모든 항목에 체크할 수 없다면? TDD를 건너뛴 것이다. 다시 시작해라.

## 막혔을 때

| 문제 | 해결 |
|------|------|
| 어떻게 테스트할지 모르겠다 | 원하는 API를 먼저 작성하라. assertion부터 써라. |
| 테스트가 너무 복잡하다 | 설계가 너무 복잡한 것이다. 인터페이스를 단순화해라. |
| 모든 걸 mock해야 한다 | 코드가 너무 결합되어 있다. 의존성 주입을 사용해라. |
| 테스트 셋업이 거대하다 | 헬퍼를 추출해라. 그래도 복잡하면 설계를 단순화해라. |

## 최종 규칙

```
프로덕션 코드 → 테스트가 존재하고 먼저 실패했어야 한다
그렇지 않으면 → TDD가 아니다
```

사용자의 명시적 허락 없이 예외는 없다.
